# NovelFusion AI · v0.2 实施方案：变体生成（MVP 闭环收尾）

> 本文档面向 Codex 实现者。前置：v0.1 已完成（上传 + 三维度分析）。
> 范围：变体生成 + 顺带清理 v0.1 留下的两项小账（`router.refresh`、结构化分析展示）。
> 不在范围：长篇 map-reduce、多本对比、多本融合、流式生成 UI。

## Context

v0.1 完成后，用户能上传小说并看到三维度分析的 `summary`，但**没有任何下游产出**——产品价值承诺（"基于分析生成变体小说"）还没兑现一次。这是 MVP 闭环最后一段，必须先打通才能拿到真实用户反馈。

同时清掉 v0.1 文档明示遗留的两项小账：

1. `analysis-panel.tsx` 缺 `router.refresh()`，分析完成后服务端卡片状态不自动同步。
2. 分析结果只展示 `summary` 字段，用户感受不到 `locations` / `characters` / `turning_points` 等结构化数据的价值——这直接影响他们是否愿意点"生成变体"。

把这两件事和变体生成放一个迭代是合适的：它们都围绕"详情页 / `/sessions/[id]`"展开，自然合并到一个 PR 不会让范围过大。

## 关键设计决策

### 变体生成

1. **触发点与位置**：在 `/sessions/[id]` 详情页，三维度卡片下方加 `<GeneratePanel>`。**只有 `session.status === "analyzed"` 时启用**，否则灰显并提示「请先完成三维度分析」。同页底部列出已生成变体列表。
2. **API 形态**：`POST /api/generate`，请求体 `{ sessionId, config: GenerateConfig }`，响应 `{ ok: true, variantId, title, wordCount } | { error }`。理由：与 `/api/analyze` 风格一致；变体生成耗时 30-60s 容易压满 Vercel Hobby 60s，但单次请求一定在窗口内（不在端点内做多步骤）。
3. **生成形式**：`generateObject` with schema `{ title: string, content: string }`，**不**用 `streamText`/`streamObject`。理由：流式 UI 复杂度大、且 v0.2 重点是验证生成质量，"等 60s 看完整结果"完全可以接受。流式留至 v0.3。
4. **LLM 输入组装**：System prompt（创作总纲）+ User prompt 拼装：
   - 用户当时三维度的 `analyses.result` 全量序列化（不再截断，因为 result 体积小，三个加起来通常 <10k token）
   - `GenerateConfig` 序列化为人类可读段落
   - 原书前 ~20,000 字符片段，让模型有"语感锚点"
5. **输出范围与 `max_tokens` 映射**：
   - `outline`：max_tokens = 2048，prompt 要求"返回章节大纲（5-10 个节点，每节点 50-150 字）"
   - `single-chapter`：max_tokens = 4096，要求"返回单一完整章节（3000-5000 字）"
   - `three-chapters`：max_tokens = 8192，要求"返回连续三章（每章 ~3000 字）"
   - 这些限制写入 prompt 但不强求模型严格遵守；UI 在变体卡上展示实际 `word_count`。
6. **标题策略**：让模型在 `generateObject` 同时返回 `title`；若为空则后端补 `"变体 · ${timestamp}"`。
7. **会话状态机扩展**：
   - 首次开始生成：`analyzed` → `generating`
   - 该次生成成功：`generating` → `done`
   - 后续再生成更多变体：状态保持 `done`，UI 用本地 loading state 表达进行中
   - 任一生成失败：状态不回退，保持 `generating` 或 `done`（取决于之前是否已有变体）
8. **多变体支持**：`variants` 表无 `unique(session_id)` 约束，天然支持一个会话多次生成。UI 按 `created_at desc` 列表展示。
9. **变体内容存储**：直接落 `variants.content` (Postgres text)。预计单变体最大 `three-chapters` ≈ 1-2 万字，Postgres TOAST 处理无压力。不另开 Storage 文件。
10. **删除与重命名**：本轮**不做**变体删除/重命名按钮——保留只读列表。删除/重命名留至 v0.3。

### v0.1 小账清理

11. **`router.refresh()` 修复**：`analysis-panel.tsx` 在每个维度成功 upsert 后调用 `router.refresh()`，确保 `session.status` 与 `analyses` 服务端读取的新状态同步到 UI；不再依赖 `useState` 单点维护。
12. **结构化分析展示**：每个维度卡新增"展开"动作（折叠态仍显示 `summary`），展开后渲染该维度的结构化字段：
    - 世界观：locations 表 + rules 列表 + power_system
    - 人物：characters 表（角色/特征/背景）+ relationships 关系列表
    - 叙事：structure/viewpoint/pacing/themes/turning_points 表 + conflicts
    - 全部组件复用 shadcn 的 Card/Badge/Separator/Accordion 等

## 当前接口与数据形态（v0.2 新增）

### `POST /api/generate`

请求体：

```json
{
  "sessionId": "uuid",
  "config": {
    "strategy": "balanced",
    "innovation": 7,
    "viewpoint": "keep",
    "style": "keep",
    "output_scope": "single-chapter",
    "extra_instructions": ""
  }
}
```

响应：

- 成功：`{ ok: true, variantId, title, wordCount }`
- 失败：`{ error: string }`

错误条件（返回 400/403/409）：

- 未登录（403）
- session 不属于当前用户（403）
- session 三维度未全部完成（409，message="请先完成三维度分析"）
- `llm_config` 不存在（409，message="请先在设置页完成兼容模型配置。"）
- LLM 调用失败（502，透传 sanitized message，**不带 API key**）

### `variants.config` JSONB 内容

形态与 `GenerateConfigSchema` 完全一致；本轮不引入额外字段。

## 端到端验证（手测）

**环境**：与 v0.1 相同。要求至少有一个会话已完成三维度分析（`session.status === "analyzed"`）。

**步骤**：

1. `npm run dev`，进入一个已完成分析的会话详情页。
   - **预期**：三维度卡片支持展开/折叠；展开后看到结构化数据（如人物表、地点表、转折点表）。
2. 三维度卡片下方出现"生成变体"表单，所有字段为 `GenerateConfigSchema` 默认值。
3. 直接提交默认值（balanced / innovation 5 / keep / keep / single-chapter / 空 extra_instructions）。
   - **预期**：表单进入 loading 态，提示"预计 30-60 秒"。`session.status` 从 `analyzed` 经过 `generating` 变为 `done`（可在 Supabase 控制台或刷新 `/sessions` 列表验证）。
   - 完成后表单下方出现一张新的变体卡：标题、徽章（策略/创新度/范围）、字数、内容预览。
4. 点变体卡的「展开」按钮，弹窗展示完整 `content`，换行保留。
5. 改 `strategy = "theme-graft"` + `innovation = 9` + `output_scope = "outline"` 再生成一次。
   - **预期**：新增第二张变体卡；卡片按 `created_at desc` 排序在最上面。
6. 故障路径：临时改坏 `llm_config.base_url`，再生成一次。
   - **预期**：toast 显示中文错误；**不在错误信息中泄露 API key 或完整 URL**；现有变体不受影响；表单恢复为可提交。
7. 权限边界：用 Browser DevTools 把请求体 `sessionId` 改成另一个用户的会话 ID 提交。
   - **预期**：403。
8. `npm run type-check` / `npm run lint` / `npm run build` 通过；可选 `node --test --import tsx src/lib/prompts/generate.test.ts`（若新增 user prompt 组装器的单测）。

## 不在本轮范围（避免范围蠕变）

- **流式生成 UI**（`streamObject` / Server-Sent Events）—— v0.3。
- **变体删除 / 重命名 / 导出**（.docx / .pdf）—— v0.3。
- **变体之间的对比视图**（diff / heatmap）—— v2.0。
- **长篇 map-reduce 深度解剖**（关系图 / 时间线 / 事件维度）—— v1.0。
- **多本上传与对比** —— v1.5。
- **多本融合生成** —— v2.5。
- **集成测试**（API 路由层）—— v1.0 与深度功能一起补。
- **prompt 工程精调**（多策略对比、温度自动选择）—— 上线收到真实反馈后再迭代。

## 给 Codex 的实施提示

- **错误返回形态** 全程统一：API 路由 `{ error: string }` + 合适 HTTP 状态；中文文案。
- **API key 安全红线**：
  - 解密只在 `getUserLLMClient` 内部发生
  - 错误响应里**不出现** `api_key`、`base_url` 完整值、原始 SDK 报错堆栈
  - 日志同上
- **`generateObject` 调用形态**：
  ```ts
  const { object, usage } = await generateObject({
    model: openai(model),
    schema: VariantResultSchema,
    system: GENERATE_SYSTEM_PROMPT,
    prompt: buildGenerateUserPrompt({ analyses, config, excerpt }),
    temperature,
    maxTokens: scopeToMaxTokens(config.output_scope),
  });
  ```
- **变体字数计算** 使用 v0.1 已有的 `countWords` 函数，**勿重复实现**。如果命名不同（如 `computeWordCount`），用现有那个。
- **`router.refresh()`** 必须在 `analysis-panel.tsx` 成功路径调用一次；fail 路径不调用（避免覆盖 toast）。
- **`GeneratePanel` 默认值** 用 `GenerateConfigSchema.parse({})` 取 zod 默认值，**不要在组件里硬编码**默认值，保证单一来源。
- **`output_scope` → `max_tokens` 映射** 集中在 `src/lib/prompts/generate.ts` 顶部一个常量对象，便于将来调整。
- **UI 文案** 全程中文；变量名 / 注释 / commit / log 英文。
- **不引入新依赖**——所需 (`react-hook-form`、`zod`、shadcn 组件) 全部已在 `package.json`。
- **章节标题正则** 与 v0.1 保持一致，避免在生成路径意外调用——本轮不消费章节切分。
- **不动数据库 schema**——`variants` 表与 `GenerateConfigSchema` 都已就位，本轮零迁移。

## 后续建议（v0.3 起步清单，本轮不做）

1. 用 `streamObject` 给变体生成加流式 UI（首 token 1-3s 出现，显著降低感知延迟）。
2. 变体删除 / 重命名 / 收藏 / 导出 `.docx` `.pdf`。
3. 详情页加"重新分析单维度"快捷入口（覆盖现有结果）。
4. 真实用户 dogfood 收集 3-5 份反馈，决定 v1.0 优先深挖 map-reduce 还是补关系图/时间线/事件维度。
5. 集成测试（API 路由层）+ prompt regression 测试（用固定输入验证输出符合 schema）。
