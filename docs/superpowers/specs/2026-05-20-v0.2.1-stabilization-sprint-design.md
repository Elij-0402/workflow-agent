## 2. 目标 / 非目标

### 目标
- 章节级分析不再每次拉整本 `cleaned_content`。
- v0.2 已存在的全部测试在 `npm test` 时执行。
- 成本估算弹窗的 ¥ 字段来自用户在 `llm_config` 中填写的价格；未填写则降级到 token 数。

### 非目标
- 不动 single ↔ dual 双轨（独立项 #2，留 v0.3 评估）。
- 不重写 blueprint PATCH 语义（独立项 #3）。
- 不引入计费历史 / 多模型价格表 / 用量审计表。**一个 user × 一个 llm_config × 一对价格**。
- 不为 OpenAI / DeepSeek 预填默认价（避免"系统替你决定"+ 跟进运营负担）。
- 不改 `chapters` / `blueprints` / `variants` 任何 schema。

## 3. 数据层：migration 0005

新文件 `supabase/migrations/0005_chapter_text_rpc_and_pricing.sql`：

```sql
-- 1) 章节切片 RPC：把 substring 推到 DB 端
create or replace function public.chapter_text(
  p_book_id uuid,
  p_start   int,
  p_length  int
) returns text
language sql
security invoker         -- 继承 books 表 RLS
stable
as $$
  select substring(cleaned_content from p_start + 1 for p_length)
  from public.books
  where id = p_book_id
    and user_id = auth.uid()
$$;
grant execute on function public.chapter_text(uuid, int, int) to authenticated;

-- 2) llm_config 价格列（CNY per 1k tokens；null 表示未填）
alter table public.llm_config
  add column if not exists input_price_per_1k_tokens  numeric(10,6) null,
  add column if not exists output_price_per_1k_tokens numeric(10,6) null;
```

### 设计要点
- RPC 用 `security invoker` —— 不绕过 RLS。`books` 表的 RLS 策略 `auth.uid() = user_id` 会先生效；函数体内的 `where user_id = auth.uid()` 是双保险。
- 价格用 `numeric(10,6)` —— 最大 9999.999999，精度 6 位小数；覆盖"100 美元每千 token + 汇率"的极端值。
- 价格列 nullable —— 旧用户全部为 null，应用层自动降级到"只显示 token 数"分支。**这就是向后兼容的全部，不需要数据回填。**
- 不加 partial unique index、不加 trigger、不加新表。
- RPC `stable` 标注 —— 不修改数据库；planner 可在同一查询内 memoize。

## 4. 服务端改动

### 4.1 `src/lib/llm/dispatch.ts`
- SELECT 多取两列：`input_price_per_1k_tokens, output_price_per_1k_tokens`。
- 返回值多两个字段：
  ```ts
  inputPricePer1k: number | null;
  outputPricePer1k: number | null;
  ```
- LLM 调用本身不读价格，只是把价格透传给上游 caller（成本估算 / 未来用量审计用）。

### 4.2 `src/app/api/analyze/chapter/route.ts`
- 删除：先 `.from('books').select('id, cleaned_content')` 再 `.slice()` 的两步。
- 替换为单次 RPC：
  ```ts
  const { data: text, error: rpcErr } = await supabase.rpc("chapter_text", {
    p_book_id: body.bookId,
    p_start: chapter.start_char,
    p_length: chapter.end_char - chapter.start_char,
  });
  if (rpcErr) return jsonError("读取章节内容失败。", 500);
  if (!text) return jsonError("当前书籍内容不可用。", 404);
  const chapterText = text;
  ```
- 错误码语义：
  - `rpcErr !== null` → 500（基础设施 / 数据库异常）。
  - `text === null` → 404（书 / 章节不存在或被删；RLS 失败也会落到这里）。
- chapter 行的 `start_char` / `end_char` 仍走原来的 `.from('chapters').select(...)` 查询取得，确保章节合法性优先校验。

### 4.3 `src/lib/llm-config.ts`
`LLMConfigFormSchema` 新增两个字段：
```ts
input_price_per_1k_tokens: z.coerce.number().min(0).max(9999).nullable().optional(),
output_price_per_1k_tokens: z.coerce.number().min(0).max(9999).nullable().optional(),
```
`parseLLMConfigFormData` 内增加空字符串归一化：`v === "" ? null : Number(v)`。
`LLMConfig` row 类型同步扩展两个字段。

### 4.4 `src/app/(app)/settings/actions.ts`
upsert payload 多写两列；明文存储；不走 AES-GCM。

### 4.5 不影响的服务端路径
- 旧 `/api/analyze`、`/api/generate`（single mode）—— 不变。
- `/api/analyze/book`、`/api/generate-v2`、`/api/blueprint*` —— 不变。
- `getUserLLMClient` 的 caller —— 现有 destructuring `{ openai, model, configId, temperature, maxTokens }` 仍工作；忽略新字段。

## 5. 客户端改动

### 5.1 `src/lib/cost/estimate.ts`
**唯一一处 API 突破**。签名变更：

```ts
export type CostBreakdown = {
  calls: number;
  estimatedInputTokens: number;
  estimatedOutputTokens: number;
  estimatedCNY: number | null;  // null = 用户未配置价格
};

export function estimateChapterBatchCost(opts: {
  chapterCount: number;
  avgCharsPerChapter: number;
  inputPricePer1k: number | null;     // 必填、显式 null
  outputPricePer1k: number | null;    // 必填、显式 null
  approxTokensPerChar?: number;       // 默认 0.6（注释说明仅为中文粗估）
  approxOutputTokensPerCall?: number; // 默认 600
}): CostBreakdown
```

- 删除 `pricePer1kInputTokens` / `pricePer1kOutputTokens` 的默认值 `0.002` / `0.006`。
- 价格为 null 时 → `estimatedCNY = null`，但 token 数仍计算并返回。
- caller 必须显式传入价格（即便是 null），避免"忘了传 → 用了一个默认数 → 假精确"的回归。

### 5.2 `src/components/workbench/cost-estimate-modal.tsx`
props 新增 `estimatedCNY: number | null` + `inputPrice / outputPrice` 用于文案。

null 分支文案：
> "未在设置中填写模型单价。本批将发起 N 次调用，估算约 X 万输入 tokens。如需金额估算，请在 [设置](/settings) 填写价格。"

非 null 分支文案：
> "按你在设置中填写的 ¥{inputPrice} / ¥{outputPrice} 每千 tokens 估算 ≈ ¥{estimatedCNY}（仅供参考，实际以 provider 账单为准）。"

链接走 `next/link` 直达 `/settings`。

### 5.3 `src/app/(app)/sessions/[id]/workbench/page.tsx`
server component 增 select：
```ts
supabase
  .from("llm_config")
  .select("input_price_per_1k_tokens, output_price_per_1k_tokens")
  .maybeSingle()
```
作为 prop `pricing: { input: number | null; output: number | null } | null` 下传 `WorkbenchClient`。

### 5.4 `src/app/(app)/sessions/[id]/workbench/workbench-client.tsx`
`askBatchConfirmation` 调 `estimateChapterBatchCost` 时把 `pricing` 透传。其他逻辑不变。

### 5.5 `src/app/(app)/settings/settings-form.tsx`
表单加两组数字输入：
- 标签 "输入 token 单价（¥/1k）" / "输出 token 单价（¥/1k）"
- `type="number"`、`step="0.000001"`、`min="0"`、可空
- placeholder 给参考格式："如 0.005"
- helper 文字：
  > "用于工作台批量分析前的成本估算，不会改变实际 LLM 调用。OpenAI / DeepSeek 的参考价请查阅各自定价页面。"

不预填具体数字（既避免"我们替你确认了价格"的心理暗示，也避免运营跟进负担）。

## 8. 风险与缓解

| 风险 | 缓解 |
|---|---|
| RPC `security invoker` 在 Supabase 中行为与预期不一致 | 手测验证："另一个 user 的 book_id 调 RPC 应返回 null"；写一条 SQL 测试在 PR 描述中贴 |
| `numeric(10,6)` 在 React Hook Form 数字输入下被字符串化 | `z.coerce.number()` 已处理；测试 case 6.2 #1–#3 覆盖 |
| 第一次跑 glob 测试暴露大批红 | sprint 范围已包含；如某测试是过时设计的产物 → 同一 PR 删除并在 commit message 说明 |
| 用户已基于硬编码 ¥ 数字做了消费决策 | 这正是要解决的；改动只让面板更诚实，不会让用户更迷茫 |
| RPC 调用相比直 SELECT 多一次 round trip | 实测 supabase-js `.rpc()` 与 `.select()` 走同一 HTTP 通道；网络 round trip 数相同 |
