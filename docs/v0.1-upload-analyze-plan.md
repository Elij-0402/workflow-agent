## 已落地的关键决策

1. **文本切片策略**
   - 清洗后的整本内容写入 `books.cleaned_content`
   - 分析时只截取前 `80,000` 字符
   - UI 文案已统一为“基于前约 8 万字符”

2. **章节元数据预留**
   - 上传时完成章节切分
   - `books.metadata` 已写入：
     - `encoding`
     - `chapters: { index, title, startChar, endChar }[]`

3. **调度形式**
   - 上传使用 Server Action：`uploadNovel(formData)`
   - 分析使用 Route Handler：`POST /api/analyze`
   - 客户端详情页并发发起 3 个维度请求

4. **错误路径**
   - 单维失败不影响其他维度
   - `analyses` 只在成功时 `upsert`
   - 当某本书 3 个维度都成功后，`session.status` 才更新为 `analyzed`

5. **编码检测**
   - 优先 `UTF-8 BOM`
   - 再尝试 `UTF-8`
   - 若 `�` 占比超过 1%，回退到 `GB18030`

6. **会话创建时机**
   - 上传时直接创建 `sessions + books`
   - `sessions.status = "uploaded"`
   - 成功后跳转到 `/sessions/{id}`

7. **存储双写**
   - 原始文件上传到 Storage：`novels/{user_id}/{session_id}/{filename}`
   - `books.storage_path` 记录完整逻辑路径
   - 清洗后文本存入 `books.cleaned_content`

8. **上传 UI**
   - 使用原生 `<input type="file">`
   - 不做拖拽、不做进度条、不做预览

9. **状态流转**
   - `uploaded -> analyzing -> analyzed`

## 当前接口与数据形态

### Server Action

`uploadNovel(formData)` 返回：

- 成功：`{ ok: true, message?, redirectTo }`
- 失败：`{ error: string }`

### Analyze API

`POST /api/analyze`

请求体：

```json
{
  "sessionId": "uuid",
  "dimension": "worldview"
}
```

响应：

- 成功：`{ ok: true, dimension, result }`
- 失败：`{ error: string }`

### `books.metadata`

当前至少包含：

```json
{
  "encoding": "utf-8 | utf-8-bom | gb18030",
  "chapters": [
    {
      "index": 1,
      "title": "第一章",
      "startChar": 0,
      "endChar": 1024
    }
  ]
}
```

## 建议继续手测

建议按下面步骤继续做真实联调：

1. 启动 `npm run dev`
2. 登录后访问 `/upload`
3. 上传一份 GB18030 编码 `.txt`
4. 确认页面跳到 `/sessions/{id}`
5. 在 Supabase 验证：
   - `books.cleaned_content` 无乱码
   - `books.metadata.chapters` 非空
   - Storage 路径为 `novels/{user_id}/{session_id}/...`
6. 在详情页点击“开始分析”
7. 确认 3 个 `POST /api/analyze` 并发发出
8. 确认单维失败时其他维度不受影响
9. 确认 3 维完成后 `session.status = analyzed`
